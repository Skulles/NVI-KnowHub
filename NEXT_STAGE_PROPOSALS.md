# Next Stage Proposals

Этот файл фиксирует предложения для следующего этапа развития `NVI KnowHub`.

## Цель следующего этапа

Перейти от локального прототипа базы знаний к рабочему контуру, в котором:
- модератор может готовить изменения в приложении;
- новая версия базы может публиковаться для всех сотрудников;
- приложение умеет безопасно получать и применять обновления;
- архитектура остается совместимой с будущей desktop-упаковкой.

## Приоритет 1

### 1. Реальный формат snapshot базы
- Утвердить рабочую `SQLite`-схему вместо текущего demo seed.
- Добавить versioning схемы и миграции.
- Разделить:
  - `official content`
  - `local drafts`
  - `user settings`
  - `search helpers / metadata`

Что сделать:
- описать таблицы `sections`, `articles`, `assets`, `drafts`, `app_meta`;
- добавить `schemaVersion`;
- предусмотреть checksum и timestamp для snapshot.

### 2. Публикация базы модератором для всех
- Добавить `publish workflow` из приложения.
- После редактирования модератор должен уметь собрать новый snapshot и отправить его в `GitHub Releases`.

Что сделать:
- подготовить export актуальной базы в `.sqlite`;
- добавить шаг валидации перед публикацией;
- загружать snapshot как release asset;
- сохранять version/tag релиза;
- показывать статус публикации и ошибки.

### 3. Безопасный доступ модератора
- Сейчас moderator mode включается локально.
- На следующем этапе стоит добавить минимальную защиту для режима публикации.

Варианты:
- локальный пароль/секрет для MVP;
- GitHub token только у модератора;
- позже переход к нормальной auth-модели.

## Приоритет 2

### 4. Надежное обновление базы у пользователей
- Сделать полноценный `update manager`.
- Приложение должно уметь не только проверять релиз, но и безопасно применять обновление.

Что добавить:
- download progress;
- checksum verification;
- backup предыдущего snapshot;
- rollback при невалидной базе;
- уведомление пользователя о новой версии.

### 5. Улучшение редактора
- Расширить `TipTap` под реальные сценарии базы знаний.

Что стоит добавить:
- таблицы;
- callout-блоки;
- вставку изображений и файлов;
- внутренние ссылки на статьи;
- шаблоны статей;
- preview перед публикацией.

### 6. Импорт и экспорт контента
- Поддержать импорт стартовой базы вне кода.
- Нужен путь, по которому knowledge base можно собирать не руками в исходниках.

Что сделать:
- импорт `.sqlite` snapshot;
- опционально импорт из `JSON/Markdown` в базу;
- экспорт локальных черновиков.

## Приоритет 3

### 7. Подготовка к Electron/Tauri
- Вынести файловые и системные операции в отдельный desktop adapter.
- Подключить локальную файловую систему для snapshot, assets и backup.

Что сделать:
- desktop-specific platform bridge;
- выбор директории хранения базы;
- открытие внешних ссылок и файлов;
- автообновление desktop-оболочки отдельно от knowledge snapshot.

### 8. Производительность и размер бандла
- Сейчас bundle уже заметно вырос из-за `sql.js` и редактора.

Что улучшить:
- lazy load для `ModeratorEditor`;
- lazy load для update/settings модулей;
- code splitting для `sql.js`;
- оптимизация initial render.

### 9. Тесты и контроль качества
- На следующем этапе полезно добавить не общие, а точечные тесты вокруг критичных сценариев.

Минимально:
- тест открытия snapshot;
- тест поиска по статьям;
- тест сохранения/удаления draft;
- тест применения обновления;
- тест, что локальные drafts не теряются после update.

## Рекомендуемый порядок работ

1. Утвердить реальную схему `SQLite`.
2. Сделать export/import snapshot.
3. Реализовать publish workflow в `GitHub Releases`.
4. Добавить безопасное применение обновлений у пользователей.
5. Расширить editor под рабочие сценарии.
6. Затем перейти к desktop-обертке.

## Что уже есть в текущем этапе

- базовый `React`-каркас;
- локальный `SQLite` snapshot через `sql.js`;
- дерево разделов, просмотр статьи, поиск;
- moderator `WYSIWYG` draft mode;
- заготовка `GitHub Releases` update flow;
- platform bridge под будущий desktop.

## Хорошая точка возврата

Когда вернетесь к проекту, логично начать с вопроса:

`Давай реализуем publish workflow для модератора: сборка SQLite snapshot и публикация новой версии в GitHub Releases.`
